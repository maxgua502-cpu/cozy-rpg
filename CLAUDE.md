# CLAUDE.md — Уютная RPG (рабочее название)

Одиночная браузерная RPG: топ-даун мир, пошаговый бой «дракон-дама-рыцарь», зверинец существ, лут-цикл, крафт, оффлайн-прогресс. Полный дизайн — в `docs/GDD.md`. **GDD — источник истины**: Реестр решений (№1-37) обязателен, отступление от решения = сначала спросить управляющего.

## Роли
- **Управляющий (Max)** — все дизайн- и продуктовые решения. Не программист: объяснять по-русски, без жаргона, предлагать варианты с trade-off'ами.
- **Claude Code** — реализация. Технические решения внутри зафиксированной архитектуры — самостоятельно.

## Стек
- Phaser 4 (плагин phaser4-gamedev: /phaser-new, /phaser-run, /phaser-validate, /phaser-build)
- Vite, vanilla JS (ES-модули). Без TypeScript в MVP.
- Vitest — юнит-тесты игровой логики. Playwright — smoke-тесты сцен (запуск, скриншот, консоль без ошибок).
- Деплой: GitHub Pages.

## Архитектура (нарушать нельзя)
1. **Логика отдельно от Phaser.** Вся игровая механика (бой, лут, прогрессия, оффлайн-расчёт) — чистые JS-модули в `src/systems/`, без импортов Phaser. Сцены Phaser — только отображение и ввод. Причина: юнит-тесты логики без браузера.
2. **Контент = данные.** Мобы, виды существ, предметы, заклинания, рецепты, квесты, зоны — JSON в `src/data/`. Новый контент не должен требовать правки кода. Тест: «добавить моба = добавить JSON».
3. **Один источник состояния.** Всё сохраняемое — в одном объекте состояния; localStorage + экспорт/импорт файла. Каждое поле состояния имеет владельца-систему.
4. **Мобайл-первый UI.** Каждый экран проверяется на 390×844 (портрет) до одобрения. Тач-цели ≥ 44px.

## Структура
```
src/
  scenes/     # Phaser-сцены: Boot, World, Battle, Camp, UI-оверлеи
  systems/    # чистая логика: combat.js, loot.js, progression.js, offline.js, crafting.js, quests.js
  data/       # JSON: creatures/, mobs/, items/, spells/, recipes/, quests/, zones/, classes.json
  ui/         # переиспользуемые UI-компоненты (панель зверинца, кукла экипировки)
assets/       # спрайты, атласы, звук
docs/GDD.md   # дизайн-документ (источник истины)
tests/        # vitest + playwright
```

## Рабочий цикл (каждая задача бэклога)
1. Прочитать задачу в GDD-бэклоге, назвать критерий verify.
2. Для логики: сначала юнит-тесты, потом реализация до зелёного.
3. `/phaser-validate` + Playwright-smoke после каждого шага.
4. Показ управляющему только после прохождения verify. Одна задача = один PR/коммит-пакет.

## Правила кода
- Простота прежде всего: минимум кода под задачу, ничего спекулятивного, никаких абстракций «на будущее».
- Хирургические изменения: не трогать соседний код, не рефакторить незапрошенное.
- Все числа баланса — в `src/data/balance.json`, не в коде (управляющий будет крутить).
- Комментарии и идентификаторы данных — на английском, тексты игры — на русском, в отдельных файлах локализации `src/data/i18n/ru.json` (задел на языки, но без i18n-библиотек в MVP).

## Что НЕ делать
- Не добавлять механики, которых нет в GDD, даже «очевидно полезные» — предлагать управляющему словами.
- Не нарушать решение №24: никаких механик, раскрывающих выбор противника в бою.
- Не вводить сервер/бэкенд/аккаунты — игра полностью локальная (решение и столп №4).
- Не оптимизировать преждевременно; не добавлять зависимости без явной необходимости (каждая новая — спросить).
- Не менять формат сейва без миграции со старых версий.

## Ключевые механики-шпаргалка (детали в GDD)
- Раунд боя: обе стороны выбирают существо (тип решает треугольник) + опц. слот действия (заклинание/зелье/свиток; у Волшебника — два). Урон = атака существа − защита существа цели, минимум 1, в HP героя. Ничья = ничего.
- Существа: виды с численностью-зарядами на бой, базовые безлимитны, восстановление после боя. Способности-проки типизированы школой, режутся резистом.
- Классы: одно уникальное правило каждому (Варвар/Волшебник/Некромант/Друид/Разбойник).
- Оффлайн: `min(прошло, потолок 8ч)` ко всем фоновым процессам при загрузке.
